微前端框架

Authors
  • Name
    Later
    Twitter

这几年微前端框架被炒作的很火，可能是之前的微服务概念火了之后大家看到了一些可行性方案了吧，以前基本上没关注过这方面的东西，直到在上佳公司离职的前几天，闲的没事干，看到了这篇文章 前端工程——基础篇， 发现公司的那个底层框架不就是一套微前端框架么，看完感触很深，思考了很多，想着如果自己从头开始实现那个微前端框架会怎么做。

文中知识点总结

文章中提到了两个观点很令人印象深刻：

  1. 前端是一种远程部署，运行时增量下载的GUI软件

  2. 前端开发最迫切需要做好的就是在基础架构中贯彻增量原则 相信这种贯彻不会随着时间的推移而改变，在可预见的未来，无论在HTTP1.x还是HTTP2.0时代，无论在ES5亦或者ES6/7时代，无论是AMD/CommonJS/UMD亦或者ES6 module时代，无论端内技术如何变迁，我们都有足够充分的理由要做好前端程序资源的增量加载。

该如何做呢，Facebook已经做了，只是没开源，但是文中总结出了微前端的核心：

静态资源管理系统 = 资源表 + 资源加载框架

资源表是一份数据文件（比如JSON），是项目中所有静态资源（主要是JS和CSS）的构建信息记录，通过构建工具扫描项目源码生成，是一种k-v结构的数据，以每个资源的id为key，记录了资源的类别、部署路径、依赖关系、打包合并等内容，比如：

{
    "a.js": {
        "url": "/static/js/a.5f100fa.js",
        "dep": [ "b.js", "a.css" ]
    },
    "a.css": {
        "url": "/static/css/a.63cf374.css",
        "dep": [ "button.css" ]
    },
    "b.js": {
        "url": "/static/js/b.97193bf.js"
    },
    "button.css": {
        "url": "/static/css/button.de33108.css"
    }
}

而资源加载框架则提供一些资源引用的API，让开发者根据id来引用资源，替代静态的script/link标签来收集、去重、按需加载资源。调用这些接口时，框架通过查表来查找资源的各项信息，并递归查找其依赖的资源的信息，然后我们可以在这个过程中实现各种性能优化算法来“智能”加载资源。

根据业务场景的不同，加载框架可以在浏览器中用JS实现，也可以是后端模板引擎中用服务端语言实现，甚至二者的组合，不一而足。

这种设计很快被验证具有足够的灵活性，能够完美支撑不同团队不同技术规范下的性能优化需求，前面提到的按需加载、延迟加载、预加载、请求合并、文件指纹、CDN部署、Bigpipe、Quickling、BigRender、首屏CSS内嵌、HTTP 2.0服务端推送等等性能优化手段都可以很容易的在这种架构上实现，甚至可以根据性能日志自动进行优化（Facebook已实现）。

因为有了资源表，我们可以很方便的控制资源加载，通过各种手段在运行时计算页面的资源使用情况，从而获得最佳加载性能。无论是前端渲染的单页面应用，还是后端渲染的多页面应用，这种方法都同样适用。

此外，它还很巧妙的约束了构建工具的职责——只生成资源表。资源表是非常通用的数据结构，无论什么业务场景，其业务代码最终都可以被扫描为相同结构的表数据，并标记资源间的依赖关系，有了表之后我们只需根据不同的业务场景定制不同的资源加载框架就行了，从此彻底告别一个团队维护一套工具的时代！

前端相比其他软件开发，在基础架构上更加迫切的需要组件化开发和资源管理，而解决资源管理的方法其实一点也不复杂：

一个通用的资源表生成工具 + 基于表的资源加载框架

实践经历

很多东西只有实践过之后才能发现问题的所在，对这篇文章印象深刻原因不仅仅是因为写的好，还是因为我们之前用对那套框架和这个思路一模一样，相当于我已经实践了四年时间了，只是我一直没有去平台那边的源码实现，不过不要紧，我觉得我可以实现这套逻辑，并将它开源出来。

简单设计

HTML

首先应该有一个基于表的资源加载框架，假设资源表和资源已经存在（这个后边再实现），并且如果有资源加载框架的话，html文件不应该加载资源了，只告诉框架应该加载哪些资源就可以了， html文件应该长成这样：

<!DOCTYPE html>
<html lang="en">
<head>
    <meta charset="UTF-8">
    <meta name="viewport" content="width=device-width, initial-scale=1.0">
    <title>Document</title>
</head>
<body>
</body>
<script src="/coreLoader.js"></script>
<script>
    fb.ready().then((loader) => {
        loader('a.js');
        loader('a.css');
    })
</script>
</html>

html先去请求资源加载框架coreLoader.js, coreLoader.js 里边去请求资源关系表，暴露出一个全局变量fb, 待一切就绪后，只需要告诉资源加载框架要加载那个资源即可， 资源加载框架loader函数会处理一切依赖关系。

资源加载框架设计

资源加载框架主要的任务就是去请求资源关系表和加载页面声明的资源和依赖的资源。

高级点的玩法

给loader函数加上事件，基于事件调用静态资源中的逻辑 loader函数添加性能分析处理，监听每个资源和资源对应的页面的性能状态
