背景

最近在阅读gitlet的源码，读到一半的时候一头雾水，又跑去读了progit内部原理部分，才豁然开朗，有一些前置的知识在这里列一下，然后再记录整个源码理解过程。

说明：gitlet是git的nodejs简化实现版本，虽然是简化，但是git的核心原理都包含在其中，我对于gitlet源码的阅读方式是用单元测试把源码全部测试一遍。

注意

这篇文章中提到的git实现基于gitlet，原理基于progit，相当于是git简化版的内部原理，和git原理大同小异。

全局例子

在初始化仓库下./test/number.txt文件内容为 first, 执行add和commit操作，此时objects目录下会生成一些文件，下边会一一解释这些文件

.git
│   HEAD
└───refs
│   └───heads
│        └───master
└───objects
│   └───1ff3c221
│   └───4bf47ca6
│   └───25d90256
│   └───41a76f91
└───index

.git 目录

在一个新目录或已有目录内执行 git init 时，Git 会创建一个 .git 目录 , 该目录下有四个重要的文件或目录： ：HEAD 及 index 文件，objects 及 refs 目录。这些是 Git 的核心部分。objects 目录存储所有数据内容，refs 目录存储指向数据 (分支) 的提交对象的指针，HEAD 文件指向当前分支，index 文件保存了暂存区域信息

例子中index内容为：test/number.txt 0 4bf47ca6

.git
│   HEAD
└───refs
│   └───heads
└───objects
└───index

git中的三个对象

存储 (数据) 对象

为每份内容生成一个文件, 取得该内容的hash值文件命名，保存在 objects 目录下，例子中 4bf47ca6 文件中的内容为 first,4bf47ca6为first字符的hash值

tree (树) 对象

tree 对象，tree 对象可以存储文件名，同时也允许存储一组文件。Git 以一种类似 UNIX 文件系统但更简单的方式来存储内容。所有内容以 tree 或 blob 对象存储，其中 tree 对象对应于 UNIX 中的目录，blob 对象则大致对应于 inodes 或文件内容。一个单独的 tree 对象包含一条或多条 tree 记录，每一条记录含有一个指向 blob 或子 tree 对象的 hash 指针，并附有该对象的类型和文件名信息。

例如：

objects目录下的 25d90256 和 1ff3c221 文件树对象

25d90256的内容为tree 1ff3c221 test， 1ff3c221 内容为 blob 4bf47ca6 number.txt

commit (提交) 对象

commit 对象都指向了你创建的树对象快照 例子中 41a76f91 文件为提交对象 内容为：

commit 25d90256
Date:  Sat Jul 24 2021 10:31:58 GMT+0800 (中国标准时间)


    commit message

Git References

refs/heads/master文件中存着当前分支最后一次提交的commit信息hash值， 例子中的内容为41a76f91,如果有其他分支的话是类似的，如下图：

每当你执行 git branch (分支名称) 这样的命令，Git 基本上就是执行 update-ref 命令，把你现在所在分支中最后一次提交的 hash 值，添加到你要创建的分支的引用

HEAD 标记

HEAD 文件是一个指向你当前所在分支的引用标识符。这样的引用标识符——它看起来并不像一个普通的引用——其实并不包含 SHA-1 值，而是一个指向另外一个引用的指针。

例子中的内容为：ref: refs/heads/master
